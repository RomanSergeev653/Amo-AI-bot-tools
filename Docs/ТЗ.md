# Задача: создать read-only инструмент OpenClaw для базы amoCRM

## Текущее состояние

В папке проекта сейчас находится только описание существующей базы данных:

```text
docs/sql
```

База PostgreSQL уже:

* создана;
* запущена;
* заполнена данными amoCRM;
* доступна по сети.

Поднимать PostgreSQL, создавать Docker-контейнер с базой, создавать таблицы, выполнять миграции или импортировать данные не нужно.

OpenClaw-бот также уже существует отдельно. Создавать нового бота, чат, интерфейс или отдельного AI-агента не нужно.

Нужно создать устанавливаемый компонент для существующего OpenClaw, который позволит безопасно читать данные из уже существующей PostgreSQL-базы amoCRM.

## 2. Цель проекта

Добавить существующему OpenClaw универсальный инструмент для выполнения безопасных SQL-запросов к базе amoCRM.

Примеры будущих вопросов:

* Сколько сейчас активных сделок?
* Сколько сделок не имеют задач?
* Сколько успешных продаж было за месяц?
* Покажи 10 последних сделок.
* Какие сделки давно не обновлялись?
* Сколько сделок находится в каждой воронке?
* Сколько продаж сделал каждый менеджер?
* Найди клиента по телефону.
* Покажи сделки определённого контакта.
* Какие задачи просрочены?
* Сравни продажи за два периода.

Не создавай отдельный tool под каждый возможный вопрос.

Нужен один универсальный инструмент, например:

```text
query_amocrm_database
```

или:

```text
run_readonly_sql
```

## 3. Архитектура

Ожидаемая схема:

```text
Пользователь
    ↓
Существующий OpenClaw
    ↓
LLM формирует SQL
    ↓
OpenClaw вызывает tool
    ↓
Tool проверяет SQL
    ↓
Tool подключается к существующей PostgreSQL-базе
    ↓
Запрос выполняется только на чтение
    ↓
Компактный результат возвращается в OpenClaw
```

Модель не должна самостоятельно устанавливать прямое соединение с PostgreSQL.

Пароль и остальные доступы должны находиться в конфигурации инструмента.

## 6. Install-скрипт

Install-скрипт должен только настроить компонент и указать доступы к существующей базе.

Он не должен создавать или запускать базу.

Во время установки нужно запросить:

```env
AMOCRM_DB_HOST=
AMOCRM_DB_PORT=5432
AMOCRM_DB_NAME=
AMOCRM_DB_USER=
AMOCRM_DB_PASSWORD=
AMOCRM_DB_SSLMODE=
```

Install-скрипт должен:

1. Проверить зависимости компонента.
2. Установить зависимости.
3. Запросить адрес существующей PostgreSQL-базы.
4. Запросить порт, по умолчанию `5432`.
5. Запросить название базы.
6. Запросить имя пользователя.
7. Запросить пароль без отображения символов.
8. Запросить SSL mode или использовать разумное значение по умолчанию.
9. Сохранить параметры в конфигурацию или `.env`.
10. Не выводить пароль в терминал.
11. Не записывать пароль в логи.
12. Проверить подключение запросом:

```sql
SELECT 1;
```

13. Проверить доступ на чтение к основным таблицам из `docs/sql`.
14. Показать понятную ошибку при неудачном подключении.
15. Зарегистрировать tool в существующем OpenClaw.
16. Вывести результат установки и способ проверки.

Install-скрипт не должен:

* устанавливать PostgreSQL;
* создавать базу;
* создавать схему;
* создавать таблицы;
* менять таблицы;
* выполнять миграции;
* загружать данные;
* создавать пользователя PostgreSQL;
* менять права пользователя базы;
* запускать Docker-контейнер базы.

## 7. Пользователь базы

Предполагается, что пользователь PostgreSQL уже создан заранее.

Компонент просто получает его логин и пароль через install-скрипт.

Желательно, чтобы этот пользователь имел права только на чтение, но инструмент не должен автоматически менять его права.

При установке можно проверить права и вывести предупреждение, если пользователь способен изменять данные.

Нельзя автоматически выполнять:

```sql
CREATE USER
GRANT
REVOKE
ALTER ROLE
```

В README можно добавить отдельную необязательную рекомендацию для администратора о создании read-only пользователя, но она не должна выполняться автоматически.

## 8. Универсальный tool

Tool должен принимать SQL и краткое описание цели запроса.

Пример интерфейса:

```json
{
  "sql": "SELECT COUNT(*) AS count FROM ...",
  "purpose": "Посчитать активные сделки"
}
```

Дополнительно можно поддержать:

```json
{
  "sql": "SELECT ...",
  "purpose": "Показать последние сделки",
  "max_rows": 10
}
```

Пользовательское значение `max_rows` не должно позволять обойти системный лимит.

---

## 9. Безопасность SQL

Доступ к базе должен быть только на чтение.

Не полагайся только на системный промпт. Выполняй программную проверку каждого запроса.

Разрешить:

```sql
SELECT
WITH ... SELECT
```

Запретить любые запросы, содержащие операции изменения данных или структуры:

```sql
INSERT
UPDATE
DELETE
TRUNCATE
DROP
ALTER
CREATE
GRANT
REVOKE
COPY
CALL
DO
VACUUM
REINDEX
REFRESH
```

Также запрети:

* несколько SQL-команд в одном вызове;
* изменение настроек сессии;
* обращение к системным функциям работы с файлами;
* выполнение программ;
* чтение файлов сервера;
* сетевые расширения;
* потенциально опасные PostgreSQL-функции.

Соединение дополнительно должно работать в read-only транзакции.

Перед выполнением запроса устанавливай ограничения:

* `statement_timeout`;
* максимальное количество возвращаемых строк;
* максимальный размер результата;
* максимальную длину SQL;
* ограничение доступных схем.

---

## 10. Ограничение объёма данных

Помощник должен стараться получать минимально необходимый объём данных.

Зафиксируй в описании tool следующие правила:

1. Сначала использовать `COUNT`, `EXISTS`, `SUM`, `AVG`, `MIN`, `MAX` и группировки.
2. Не получать полный список сущностей, когда достаточно количества.
3. Не использовать `SELECT *`.
4. Для списков по умолчанию возвращать не более 10 строк.
5. Для полной информации сначала определить конкретный ID.
6. Не выгружать всю таблицу.
7. Не получать персональные данные без необходимости.
8. Использовать только поля, необходимые для ответа.

На стороне backend обязательно установи собственный жёсткий предел строк, независимо от SQL модели.

## 11. Результат запроса

Tool должен возвращать компактный структурированный результат.

Пример успешного ответа:

```json
{
  "success": true,
  "columns": ["count"],
  "rows": [
    {
      "count": 37
    }
  ],
  "row_count": 1,
  "truncated": false,
  "execution_time_ms": 24
}
```

Пример ошибки:

```json
{
  "success": false,
  "error_type": "query_rejected",
  "message": "Разрешены только SELECT-запросы"
}
```

Не возвращай модели:

* пароль;
* строку подключения;
* внутренний stack trace;
* конфигурацию окружения;
* лишние диагностические сведения.
